# Arena — 프로젝트 방향성

## 한 줄 요약

> **5~20인 소규모 팀은 카톡으로 버티다 파일 유실, 기록 부재, 법적 리스크(주52시간)에 노출되어 있고,
> 기존 그룹웨어는 과하고, 여러 툴 조합은 번거롭다.
> 메신저 기반에 공지확인 + 간단결재 + 출퇴근기록만 네이티브로 붙인 모바일 앱이 이 공백을 채울 수 있다.**

---

## 문제 인식

소규모 업장(카페, 매장, 스튜디오, 공방, 소규모 사무실 등)은 그룹웨어가 필요하지만 도입이 어렵다.

- 기존 그룹웨어(Notion, Slack, 다우오피스 등)는 기능이 과하거나, 세팅이 복잡하거나, 비용이 부담
- 현실적으로 카카오톡 단톡방 + 엑셀 + 구글 캘린더 조합으로 버티고 있음
- 정보가 흩어지고, 나중에 찾기 어렵고, 업무와 잡담이 섞임

**Arena는 이 틈새를 파고든다.** "그룹웨어를 쓸 만큼 크진 않지만, 관리할 도구는 필요한" 조직을 위한 서비스.

### 협업 도구가 아닌 관리 앱

Slack, Notion 같은 서비스는 "같이 일하는 도구(협업)"다. Arena는 "조직을 운영하고 관리하는 도구"를 지향한다. 소규모 업장의 소유자/운영자 관점에서, 팀원과 협업하는 것보다 조직을 돌리고 정보를 관리하는 것이 더 절실한 문제다.

### 데이터로 보는 소규모 팀의 현실

오픈서베이 업무 툴 트렌드 리포트에 따르면:

- 9인 이하 소규모 회사 직장인의 **65.3%가 업무용 메신저로 카카오톡 사용**
- 9인 이하 회사에서 협업/문서 공유 툴을 **사용하지 않는 비율 65.2%**
- 100인 이상 기업의 자체 메신저 사용률 43.9% vs 9인 이하는 **7.5%에 불과**
- 업무 툴 만족 이유: "사용이 간단하고 쉽다" 67.6%, "익숙하다" 66.2%

### 카카오톡으로 업무할 때의 페인 포인트

- **파일 만료**: 공유 파일이 약 1개월 후 만료, 나중에 참여한 사람은 열람 불가
- **히스토리 유실**: 채팅방 나가면 대화 내역 전부 삭제
- **공사 구분 불가**: 개인 채팅과 업무 채팅이 섞임
- **보안 취약**: 퇴사자가 채팅방에 남아 회사 정보에 계속 접근 가능
- **멘션/확인 불가**: 특정인 지목 메시지 불가, 읽음 확인 미지원

### 왜 전환이 안 되는가?

- **익숙함의 관성**이 엄청나게 큼 (카톡 UX에 대한 의존)
- 슬랙/잔디 도입 시 **학습 비용** 발생
- 기존 그룹웨어는 소규모에게 **과하고 비쌈**
- 여러 툴 조합 시 **작업 전환 비용 2~3배 증가** (연구 결과)
- 소규모 팀은 **유료 도구에 돈 쓰는 습관 자체가 약함**

## 시장 기회

### 국내 협업 SW 시장

- 2025년 약 **2,400억 원** 규모 (Statista)
- 2020년 이후 연평균 **두 자릿수 성장률** 지속
- 코로나 이후 기업들이 협업 SW 효과를 체감하면서 수요 지속 전망

### 주요 플레이어 (500인 미만 기업 기준)

| 순위 | 제품 | 점유율 |
|------|------|--------|
| 1 | 하이웍스 (가비아) | 약 22% |
| 2 | 카카오워크 | 약 12% |
| 3 | 비즈메카 (KT) | 약 10% |
| 4 | 네이버웍스 | 약 10% |
| 5 | U+웍스 (LG유플러스) | 약 8% |

### 시장 트렌드

- 그룹웨어와 협업툴의 경계가 무너지고 **올인원 형태**로 진화 중
- AI 기능 도입 경쟁 (잔디-클로드 연동, 노션 AI 등)
- 기존 그룹웨어 대부분 **PC 웹 기반**에서 출발 → 모바일은 부가적 위치
- UI 올드하다는 평가가 많은 제품들 (메일플러그 등)

### 글로벌 유사 사례: The Mobile-First Company

- 2025년 10월 Base10 + Lightspeed로부터 **$12M 시드 유치**
- 소규모 팀을 위한 모바일 퍼스트 AI 업무 도구
- 핵심 메시지: "대부분의 도구가 대기업이나 엔지니어를 위해 만들어져 있다"
- 투자자들이 이 테제에 돈을 넣고 있음 → **시장 기회 자체는 인정받는 중**

### Arena의 포지셔닝

- 카톡 → 너무 개인용, 업무 기능 없음
- 슬랙/잔디 → IT 기업 중심, 소규모 현장직에게 과함
- 기존 그룹웨어 → PC 기반, 기능 과다, 가격 부담
- **Arena** → 모바일 퍼스트, 메신저 기반, 딱 필요한 업무 기능만

## 핵심 원칙

### 메신저가 중심이다

- 소규모 조직에서 가장 많이 쓰는 도구는 메신저
- 메신저가 메일 + 게시판의 역할을 상당 부분 흡수해야 한다
- "흘러가는 대화"와 "남겨야 하는 정보"를 구분할 수 있어야 한다

### 단순함을 유지한다

- 5~20명 규모에 불필요한 복잡성을 넣지 않는다
- 세팅 없이 바로 쓸 수 있어야 한다
- 플러그인 범위는 최소로 시작하되, 코어 기능의 완성도는 타협하지 않는다

### 모바일 퍼스트

- 타겟 사용자 대부분은 데스크톱 앞에 앉아있지 않다
- 모든 핵심 기능은 모바일에서 자연스럽게 동작해야 한다
- PC 그룹웨어를 모바일에 우겨넣는 것이 아니라, 모바일에서 자연스러운 형태로 풀어낸다
- 기존 제품 대부분이 웹뷰 래핑 수준 → **네이티브 경험의 품질**이 차별화

## 기능 전략

### 기능 우선순위

Arena는 모든 기능을 한꺼번에 제공하지 않는다. 코어 기능을 기본 제공하고, 나머지는 필요에 따라 붙여서 사용한다. 이 구조로 "기능이 과하다"는 기존 그룹웨어의 문제를 구조적으로 해결한다.

#### 필수 — 메신저 (카톡 대체)

Arena의 핵심. 업무/개인 분리, 이전 대화 열람 가능, 퇴사자 관리.

- 1:1 DM, 그룹 채팅
- 파일/이미지 전송 (**만료 없이 영구 보관 + 검색** — 카톡 대비 핵심 차별점)
- 읽음 확인, 푸시 알림
- 메시지 검색, 오프라인 지원

#### 필수 — 공지/확인

전통적 게시판은 이 규모에서 **안 쓴다** (올리는 사람도, 보는 사람도 없음). 대신 채팅 내 **공지 고정 + 읽음 확인 추적**이면 충분하다. 카톡 대비 차별점: 공지가 대화에 묻히지 않는다.

#### 높은 가치 — 간단한 승인 (전자결재 라이트)

대기업식 3단계 결재라인이 아닌, **요청 → 승인/반려 → 기록 자동 누적** 흐름만 있으면 된다.

우선 지원할 유형:
- **휴가/연차 신청**: 법적으로도 기록 필요 (연차 사용 촉진 의무, 미사용 연차 수당 분쟁)
- **경비 청구**: 영수증 사진 + 금액 + 승인 → 월말 정산 간소화

핵심 가치는 결재 프로세스가 아니라 **히스토리가 남는 것**이다. 메신저 안에서 자연스럽게 동작하면 전환 장벽이 최소화된다. (예: 채팅에서 "휴가 신청" 버튼 → 양식 → 승인 시 팀 채널 자동 공지)

#### 높은 가치 — 출퇴근 기록 (근태관리)

"5~20인인데 필요할까?" → **법적으로 필수다.**

- 2025년 1월부터 5인 이상 30인 미만 사업장에 **주 52시간제 계도기간 종료**
- 위반 시 사업주에게 2년 이하 징역 또는 2천만 원 이하 벌금
- 아직 엑셀이나 수기로 관리하는 소규모 사업장이 대부분
- 모바일 GPS 기반 출퇴근 한 번 찍으면 자동 기록 → 이것만으로 가치
- **"안 쓰면 불편한" 수준이 아니라 "안 쓰면 법 위반"** → 강력한 도입 동기

#### 차별화 — 파일 영구 보관 + 검색

별도 파일 관리 시스템은 불필요 (구글 드라이브 등 이미 존재). 대신 **메시지에 첨부된 파일이 만료되지 않고 검색 가능**한 것이 카톡 대비 가장 체감되는 차별점이다.

#### 불필요

- **전통적 게시판**: 공지 채널로 대체
- **메일**: 5~20인은 개인 메일이나 Gmail 사용
- **화상회의**: Zoom/Google Meet로 충분, 직접 만들 필요 없음

### 기능 간 시너지

```
근태관리 (출퇴근 기록)
    ↕ 연동
간단 결재 (휴가 → 연차 잔여일수 자동 차감)
    ↕ 연동
메신저 (승인 알림, 팀 공지 자동 게시)
    ↕ 연동
파일 보관 (경비 영수증, 결재 문서 자동 아카이빙)
```

→ "이 사람이 이번 달 몇 시간 일했고, 연차 몇 개 썼고, 경비 얼마 청구했다"가 한 화면에서 보임
→ 별도 HR 솔루션 도입 여력 없는 소규모 팀에게 큰 가치

## MVP

코어(메신저 + Org/Team 관리)를 완성도 높게 구현하는 것이 MVP다.

"적은 수의 사용자"가 대상이지, 기능 자체를 적게 만드는 것이 아니다. 메신저는 기능이 부족하면 사람들이 불편을 느끼므로, 카카오톡에 준할 정도의 완성도를 갖춰야 한다.

### 메신저

- 1:1 DM, 그룹 채팅 (채널형 대화방은 성격 정의 후 별도 진행)
- 파일/이미지 전송
- 읽음 확인
- 푸시 알림
- 메시지 검색
- 오프라인 지원

### Org/Team 관리

- Org 생성 및 멤버 초대 (초대 코드)
- Org별 프로필 관리
- 하위 Team 관리
- 권한 관리 (소유자/사용자)

## 계정 및 조직 구조

### 계정과 프로필 (Profile 기반 시스템)

- **User**는 인증/가입용. Supabase uid와 1:1 매핑. 이후 모든 활동에는 관여하지 않음
- **Profile**이 실제 활동 주체. 모든 조직 내 활동(팀, 대화방, 메시지 등)은 profileId 기준
- 가입 시 기본 프로필(orgId=null) 생성. Org 가입 시 Org 전용 프로필 자동 생성
- 하나의 계정으로 여러 Org에 참여 가능 (예: 알바생이 카페 2곳에서 근무)
- Org간 프로필은 완전 독립 (이름, 아바타, 역할 각각 다름)
- Org 멤버십 = Org 프로필 자체. 별도 멤버십 테이블 없이 Profile이 멤버십을 겸함

### Org

Org는 Arena의 최상위 단위다. 회사, 단체, 매장 등이 Org를 생성한다.

- 소유자(사장, 대표 등)가 Org를 생성하고 사람들을 초대
- Org 가입은 **명시적 초대** 또는 **초대 코드**를 통해서만 가능
- 공개 범위 설정 가능 (검색 노출 / 비공개)
  - 초기에는 비공개를 기본으로, 공개 범위 설정은 추후 구현

### Team

Org 내부의 하위 그룹이다. (예: 카페 Org 안에 "홀 팀", "주방 팀" 등)

- 플랫 구조 (1단계): 계층형 조직도는 소규모에서 불필요
- 필요시 확장 가능하도록 설계하되, 초기에는 단순하게 유지

### 권한 모델

- **소유자**: Org 생성자. 모든 관리 권한
- **사용자**: 일반 구성원. 기본 기능 사용

> 관리자 역할은 초기에는 두지 않는다. 소유자가 특정 사용자에게 일부 권한을 위임하는 형태는 필요시 추가.

## 배포 및 플랫폼

### 서비스 형태

- **SaaS**: 클라우드 배포, 가입 후 바로 사용
- 소규모 업장이 서버를 직접 관리할 필요 없음

### 클라이언트

- **모바일 앱 (주력)**: React Native / Expo
- **웹 (관리용)**: 모바일에서 조작하기 불편한 기능을 웹으로 제공. 멤버 관리부터 시작. 별도 웹앱

### 백엔드

- API 서버는 모바일/웹 공유
- 기존 Arena에서 검증된 패턴 재사용 (오프라인 퍼스트, 이벤트 기반 동기화 등)

### 레포 구조

- 모노레포: 백엔드(Spring Boot) / 프론트엔드(React Native) / 웹(관리용) 통합

## 수익 모델

### 기본 방향

- **프리미엄(Freemium)**: 메신저 무료 + 근태/결재 기능 유료
- 소규모 팀이 유료 도구에 돈 쓰는 습관이 약하므로, 무료로 진입 → 가치 체감 후 전환 유도
- 플러그인 구조와 자연스럽게 연결 (기본 플러그인 무료, 고급 플러그인 유료)

### 과금 기준

- 인원수 기반보다는 **기능/사용량 기반 과금** 방향
- 스토리지 사용량, 서버 운영 비용 등 실제 비용이 발생하는 지점에서 과금

## 리스크 & 고려사항

### 경쟁 리스크

- 카카오워크가 카톡 익숙함을 무기로 같은 시장 공략 중
- 하이웍스 무료 플랜 (오피스 메신저 + 전자결재 + 게시판) 존재
- 차별화 핵심: **모바일 네이티브 경험의 품질** (기존 제품 대부분 웹뷰 래핑 수준)

### 전환 장벽

- 카톡 관성이 가장 큰 적
- "팀 전원이 동시에 옮겨야" 하는 네트워크 효과 문제
- 온보딩을 극도로 쉽게 만들어야 함 (연락처 기반 초대 등)

## 기존 Arena에서 가져갈 것

### 패턴 및 설계

- SQLite 오프라인 퍼스트 아키텍처
- 이벤트 기반 UI 갱신 (intent 패턴)
- DB 레이어 패턴 (스키마 파일 구조, upsert 트랜잭션 규칙)
- WebSocket 실시간 통신

### 코딩 규칙 (프론트엔드)

- 인라인 구현 우선, 섣부른 추상화 금지
- 공통 함수 추출은 명시적 요청 시에만
- 명시적이고 예측 가능한 API 설계

> Spring Boot 백엔드 코딩 규칙은 별도 정리 예정.

### 기술 스택

- 프론트엔드: React Native + Expo
- 백엔드: Spring Boot
- DB: PostgreSQL + SQLite (클라이언트)
- 인증: Supabase Auth
- 인프라: 결정 필요
