# Arena — 프로젝트 방향성

## 한 줄 요약

소규모 조직(5~20명)을 위한 경량 그룹웨어. 모바일 메신저를 중심으로 업무에 필요한 기능을 가볍게 제공한다.

---

## 문제 인식

소규모 업장(카페, 매장, 스튜디오, 공방, 소규모 사무실 등)은 그룹웨어가 필요하지만 도입이 어렵다.

- 기존 그룹웨어(Notion, Slack, 다우오피스 등)는 기능이 과하거나, 세팅이 복잡하거나, 비용이 부담
- 현실적으로 카카오톡 단톡방 + 엑셀 + 구글 캘린더 조합으로 버티고 있음
- 정보가 흩어지고, 나중에 찾기 어렵고, 업무와 잡담이 섞임

**Arena는 이 틈새를 파고든다.** "그룹웨어를 쓸 만큼 크진 않지만, 관리할 도구는 필요한" 조직을 위한 서비스.

### 협업 도구가 아닌 관리 앱

Slack, Notion 같은 서비스는 "같이 일하는 도구(협업)"다. Arena는 "조직을 운영하고 관리하는 도구"를 지향한다. 소규모 업장의 소유자/운영자 관점에서, 팀원과 협업하는 것보다 조직을 돌리고 정보를 관리하는 것이 더 절실한 문제다.

---

## 핵심 원칙

### 메신저가 중심이다

- 소규모 조직에서 가장 많이 쓰는 도구는 메신저
- 메신저가 메일 + 게시판의 역할을 상당 부분 흡수해야 한다
- "흘러가는 대화"와 "남겨야 하는 정보"를 구분할 수 있어야 한다

### 단순함을 유지한다

- 5~20명 규모에 불필요한 복잡성을 넣지 않는다
- 세팅 없이 바로 쓸 수 있어야 한다
- 플러그인 범위는 최소로 시작하되, 코어 기능의 완성도는 타협하지 않는다

### 모바일 퍼스트

- 타겟 사용자 대부분은 데스크톱 앞에 앉아있지 않다
- 모든 핵심 기능은 모바일에서 자연스럽게 동작해야 한다
- PC 그룹웨어를 모바일에 우겨넣는 것이 아니라, 모바일에서 자연스러운 형태로 풀어낸다

---

## 기능 범위

### 구조: 코어 + 플러그인

Arena는 모든 기능을 한꺼번에 제공하지 않는다. 코어 기능(메신저 + Org/Team 관리)은 기본 제공하고, 나머지 기능은 플러그인으로 조직이 필요에 따라 붙여서 사용한다.

이 구조로 "기능이 과하다"는 기존 그룹웨어의 문제를 구조적으로 해결한다.

### 코어 (기본 제공)

| 기능 | 역할 |
|------|------|
| **메신저** | 실시간 대화, 알림 허브, 정보 전달의 중심축 |
| **Org/Team 관리** | 멤버, 권한, Org/Team 구조 |

### 플러그인 (필요시 활성화)

| 플러그인 | 역할 | 구현 순서 |
|------|------|----------|
| **게시판/공지** | 남겨야 하는 정보 관리, 공지사항 | 1 |
| **업무 관리** | 할일, 업무 배분, 진행 상황 추적 | 2 |
| **캘린더** | 일정 공유, 근무표, 기한 관리 | 3 |

> 플러그인 목록은 업종별 니즈 분석을 통해 계속 확장한다. (→ 업종별 니즈 문서 별도 작성 예정)

### 기능 간 연결 (추후)

- 메시지 → 공지로 고정
- 메시지 → 할일로 전환
- 할일 기한 → 캘린더에 표시
- 게시판/할일 변경 → 메신저 알림

> 초기에는 각 기능을 독립적으로 구현하고, 연결은 이후에 추가한다. 기능 간 연결은 해당 플러그인이 활성화된 경우에만 동작한다.

---

## MVP

코어(메신저 + Org/Team 관리)를 완성도 높게 구현하는 것이 MVP다.

"적은 수의 사용자"가 대상이지, 기능 자체를 적게 만드는 것이 아니다. 메신저는 기능이 부족하면 사람들이 불편을 느끼므로, 카카오톡에 준할 정도의 완성도를 갖춰야 한다.

### 메신저

- 1:1 채팅, 그룹 채팅, 채널 기반 채팅
- 파일/이미지 전송
- 읽음 확인
- 푸시 알림
- 메시지 검색
- 오프라인 지원

### Org/Team 관리

- Org 생성 및 멤버 초대 (초대 코드)
- Org별 프로필 관리
- 하위 Team 관리
- 권한 관리 (소유자/사용자)

---

## 계정 및 조직 구조

### 계정

- 개인 가입 후, Org에 별도로 가입하는 구조
- 하나의 계정으로 여러 Org에 참여 가능 (예: 알바생이 카페 2곳에서 근무)
- Org에 가입하는 순간 해당 Org 전용 프로필이 분리 생성됨 (Org간 프로필 독립)

### Org

Org는 Arena의 최상위 단위다. 회사, 단체, 매장 등이 Org를 생성한다.

- 소유자(사장, 대표 등)가 Org를 생성하고 사람들을 초대
- Org 가입은 **명시적 초대** 또는 **초대 코드**를 통해서만 가능
- 공개 범위 설정 가능 (검색 노출 / 비공개)
  - 초기에는 비공개를 기본으로, 공개 범위 설정은 추후 구현

### Team

Org 내부의 하위 그룹이다. (예: 카페 Org 안에 "홀 팀", "주방 팀" 등)

- 플랫 구조 (1단계): 계층형 조직도는 소규모에서 불필요
- 필요시 확장 가능하도록 설계하되, 초기에는 단순하게 유지

### 권한 모델

- **소유자**: Org 생성자. 모든 관리 권한
- **사용자**: 일반 구성원. 기본 기능 사용

> 관리자 역할은 초기에는 두지 않는다. 소유자가 특정 사용자에게 일부 권한을 위임하는 형태는 필요시 추가.

---

## 배포 및 플랫폼

### 서비스 형태

- **SaaS**: 클라우드 배포, 가입 후 바로 사용
- 소규모 업장이 서버를 직접 관리할 필요 없음

### 클라이언트

- **모바일 앱 (주력)**: React Native / Expo
- **웹 (관리용)**: 모바일에서 조작하기 불편한 기능을 웹으로 제공. 멤버 관리부터 시작. 별도 웹앱

### 백엔드

- API 서버는 모바일/웹 공유
- 기존 Arena에서 검증된 패턴 재사용 (오프라인 퍼스트, 이벤트 기반 동기화 등)

### 레포 구조

- 모노레포: 백엔드(Spring Boot) / 프론트엔드(React Native) / 웹(관리용) 통합

---

## 수익 모델 (초기 방향)

지금 당장 구체화하지는 않으나, 방향성만 정리해둔다.

- 인원수 기반보다는 **기능/사용량 기반 과금** 방향
- 스토리지 사용량, 서버 운영 비용 등 실제 비용이 발생하는 지점에서 과금
- 플러그인 구조와 자연스럽게 연결될 수 있음 (기본 플러그인은 무료, 고급 플러그인은 유료 등)

---

## 기존 Arena에서 가져갈 것

### 패턴 및 설계

- SQLite 오프라인 퍼스트 아키텍처
- 이벤트 기반 UI 갱신 (intent 패턴)
- DB 레이어 패턴 (스키마 파일 구조, upsert 트랜잭션 규칙)
- WebSocket 실시간 통신

### 코딩 규칙 (프론트엔드)

- 인라인 구현 우선, 섣부른 추상화 금지
- 공통 함수 추출은 명시적 요청 시에만
- 명시적이고 예측 가능한 API 설계

> Spring Boot 백엔드 코딩 규칙은 별도 정리 예정.

### 기술 스택

- 프론트엔드: React Native + Expo
- 백엔드: Spring Boot
- DB: PostgreSQL + SQLite (클라이언트)
- 인증: Supabase Auth
- 인프라: 결정 필요
